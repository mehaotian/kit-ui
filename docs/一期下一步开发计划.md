## 3. 迭代 2：展示与反馈闭环（主战场，建议 1~2 周）

### 3.1 迭代目标

解决业务页面「列表承载 + 交互反馈」主链路，使表单提交后的结果展示、确认操作、轻提示具备最小可用能力。

### 3.2 组件任务（按推荐顺序）

#### 第一梯队（强依赖，先做）

| 顺序 | 组件 | 预估 | 关键能力 | 依赖 |
| --- | --- | --- | --- | --- |
| 1 | `k-popup` + `k-overlay` | 2~3 天 | 五方向弹出层、遮罩、关闭策略 | — | ✅ v1 三端验收 |
| 2 | `k-toast` | 1~2 天 | 轻提示、命令式 API、自动消失 | k-overlay | ✅ v1 |
| 3 | `k-modal` | 2~3 天 | 确认/取消、beforeClose、复用 k-popup | k-popup | ✅ v1 |

#### 第二梯队（列表承载）

| 顺序 | 组件 | 预估 | 关键能力 | 依赖 |
| --- | --- | --- | --- | --- |
| 5 | `k-list` v2 | 1~2 天 | 三态状态壳 + k-cell 行 | — | ✅ v2 已实施 |

> **说明**：现有 `k-cell` / `k-cell-group` 已覆盖部分列表场景；`k-list` 应定位为更通用的数据列表容器，避免与 cell 能力完全重复，可在设计阶段明确差异（如虚拟滚动预留、批量操作 slot 等）。

#### 并行增强（可穿插）

| 组件 | 预估 | 说明 |
| --- | --- | --- |
| `k-swiper` | 2~3 天 | 轮播图，首页/ Banner 场景 |
| `k-swiper-item` | 0.5~1 天 | 轮播项，与 swiper 同批交付 |

### 3.3 每个组件默认交付物

沿用 `docs/一期组件迭代看板.md` Issue 模板：

- [ ] 组件 API（props / emits / slots）
- [ ] demo 页面 + `pages.json` 注册
- [ ] README 与实现一致
- [ ] APP / WEB / MP 冒烟验证
- [ ] 满足 kit-ui 一期 rules 红线（无 grid/gap/@media）

### 3.4 迭代 2 DoD（完成标准）

- [ ] 列表场景可承载真实数据项（空态、长内容、边界项）
- [x] 弹层与提示具备最小可用交互（开关、关闭、禁用态）
- [x] 表单提交 → modal 确认 → toast 反馈可串联 demo（`pages/modal/modal.uvue`）
- [x] popup / toast / modal 示例文案与组件行为一致
- [x] 反馈类首页入口为 `completed`（List 仍 planning）

### 3.5 技术预研要点（开发前 0.5 天）

| 主题 | 关注点 |
| --- | --- |
| APP 动画 | popup/modal 开合动效走 `kit-ui-app-animation` 技能，禁止响应式 `:style` 驱动 |
| 命令式 API | toast 需评估 UTS 下全局挂载方案（provide/inject vs 单例组件） |
| 层级管理 | popup/modal/toast 的 z-index 与遮罩叠加策略 |
| 与 k-form 联动 | modal 内嵌表单、提交后 toast 反馈 — demo 预留组合示例 |

## 8. 风险与决策点

| 风险 | 应对 |
| --- | --- |
| popup/modal 在 APP 端动画不生效 | 提前读 `kit-ui-app-animation`，首周做 APP 验证 |
| k-list 与 k-cell 能力重叠 | 开发前写 1 页差异说明，避免重复建设 |
| 命令式 toast 在 UTS 下难实现 | 备选方案：声明式 `<k-toast>` + ref 调用 |
| 产能分散 | 迭代 2 主任务不超过 5 个组件；picker 等延后到迭代 3 间隙 |
