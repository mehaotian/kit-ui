- 统计时间：2026-06-18

## 3. 高优先级重排（建议按批次推进）

## 批次 A（必须先做，补齐最短板）

- `k-input`（先收敛为可发布版本）
- `k-textarea`
- `k-radio` / `k-checkbox` / `k-switch`
- `k-form` / `k-form-item`

原因：表单基础能力缺失会直接阻断业务页面接入，优先级高于新增视觉类组件。

## 批次 B（页面结构与反馈闭环）

- `k-list` / `k-list-item`
- `k-popup` / `k-modal` / `k-toast`

原因：列表承载内容，弹层与提示承载交互反馈，是业务页面的第二层基础设施。

## 批次 C（导航能力）

- `k-tabs` / `k-tab-pane`
- `k-navbar`
- `k-tabbar` / `k-tabbar-item`

原因：导航组件通常依赖前两批组件能力，放在第三批可减少返工。

## 批次 D（增强与体验）

- `k-swiper` / `k-swiper-item`
- `k-pull-refresh` / `k-load-more`
- `k-empty` / `k-skeleton` / `k-progress`（按需求挑选）

原因：属于“体验增强”，可在核心能力稳定后并行推进。

## 4. 目前规划中“不合理/不建议一期做”的组件

## 4.1 强业务耦合，不适合作为基础 UI 库一期目标

- `k-goods-card`、`k-goods-list`、`k-goods-detail`
- `k-cart`、`k-submit-bar`
- `k-address-picker`、`k-address-list`、`k-address-edit`
- `k-user-card`、`k-comment`

建议：迁移到“业务组件扩展包”或示例仓库，不占用一期核心产能。

## 4.2 高复杂 + 平台限制明显，建议二期评估

- 图表类：`k-line-chart`、`k-bar-chart`、`k-pie-chart`、`k-doughnut-chart`
- 地图类：`k-map`、`k-map-marker`
- 富文本编辑类：`k-editor`、`k-markdown`

建议：先做调研与技术预研，不作为一期承诺交付。

## 4.3 性价比较低或能力重复，建议不单独做组件

- `k-link`（可由 `k-text` + 点击能力覆盖）
- `k-copy`、`k-share`（更适合沉到 `utils`）
- `k-scroll-view`（平台原生能力已成熟，封装收益低）

## 4.4 清单命名质量问题（需治理）

- `k-menk-item`（疑似拼写错误）
- `k-skk-selector`（疑似拼写错误）

建议：统一命名规范后再决定是否进入研发排期。

## 5. 一期目标建议（收敛版）

将“一期组件池”控制在 **30~40 个**，按“可复用基础能力”优先，暂不追求“全场景全覆盖”。

建议一期范围：

- 基础元素与展示：按钮、图标、文本、图片、标签、徽标、分割线、卡片、单元格、折叠、时间轴、加载、头像、宫格、布局
- 表单核心：输入、文本域、单选、复选、开关、表单容器
- 交互反馈：弹层、模态、轻提示
- 导航核心：标签页、导航栏、底栏
- 列表基础：列表、列表项

## 6. 未来 3 个迭代开发计划（建议）

- 迭代 1（1~2 周）：完成批次 A，关闭 `k-input` 历史文件（copy/backup）问题，补齐文档与 demo。
- 迭代 2（1~2 周）：完成批次 B，补齐最小可用的页面搭建与反馈闭环。
- 迭代 3（1~2 周）：完成批次 C，并启动批次 D 中需求最高的 2~3 个组件。

## 7. 治理建议（与本次盘点配套）

- 将 `docs/组件开发清单.md` 的“开发状态”从“统一待开发”改为真实状态（已完成/进行中/规划中/不纳入一期）。
- 首页状态、`pages.json`、组件目录三者建立单一事实源，避免多处状态不一致。
- 对 `copy`、`backup` 等临时文件设定治理规则：进入可发布状态前必须清理。
