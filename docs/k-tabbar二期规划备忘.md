# k-tabbar 二期规划备忘

> 日期：2026-06-29  
> 状态：**不纳入一期**；方案待二期讨论  
> 决策人：产品 / 架构（用户确认）

---

## 1. 一期不做的原因

| 维度 | 说明 |
| --- | --- |
| 平台内置能力 | uni-app / 各端 **原生 tabBar** 由 `pages.json` + 原生容器实现，应用级路由切换 |
| 自定义成本高 | 完全自定义 tabBar 需处理：安全区、路由同步、选中态、图标字体、各端导航栈差异 |
| 与 k-tabs 易混淆 | `k-tabs` = 页面内分区；tabBar = 应用级底栏，API 与视觉不应合并 |
| 产能 | 一期迭代 3 聚焦 `k-tabs` + `k-navbar`，tabbar 独立专项更合适 |

**结论**：首页 `Tabbar` 入口保持 `planning`，一期不交付 `k-tabbar` / `k-tabbar-item` 组件。

---

## 2. 二期需讨论的方案方向

### 方案 A：文档 + 配置引导（轻量）

- 不封装组件，提供 **pages.json tabBar 配置模板** + 主题色与 kit-ui 变量对齐说明
- 适合：标准三栏/五栏底栏，无复杂交互

### 方案 B：半自定义（推荐评估）

- 使用 uni-app x **自定义 tabBar** 能力（若目标端 API 稳定）
- kit-ui 提供：`k-tabbar` 仅负责 **视觉层**（图标、文字、选中态、安全区）
- 路由切换由页面 `onTabItemTap` / 官方自定义 tabBar 协议驱动

### 方案 C：全自定义浮层

- `position: fixed` 底栏 + 自行维护 `current` 与 `uni.switchTab` / `reLaunch`
- 灵活但易与原生返回栈、安全区、键盘顶起冲突

### 待确认问题清单

1. 一期目标端（微信小程序 / APP / WEB）自定义 tabBar 官方支持度与限制？
2. 是否需要 **凸起按钮**、**中间大图标** 等异形底栏？
3. 是否与 `k-navbar` 联动（隐藏系统 tabBar 的全屏页）？
4. 图标是否统一走 `k-icon`，选中/未选中双色如何主题化？
5. 是否需要与路由库（如 uni-router）集成示例？

---

## 3. 一期替代方案（业务侧）

- 应用级底栏：继续使用 **`pages.json` 原生 tabBar**，主色在 `globalStyle` / 原生配置中与 `#6366f1` 对齐
- 页面内切换：使用 **`k-tabs`**（本期开发）
- 临时多页导航：多个普通页面 + `uni.navigateTo`，或 `k-navbar` 返回
