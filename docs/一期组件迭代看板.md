# kit-ui 一期组件迭代看板（可直接拆 Issue）

## 使用说明

- 本看板基于 `docs/组件现状盘点与优先级规划.md` 与 `docs/组件开发清单.md`。
- 迭代目标按“先可用、再完善”的原则推进：先补齐核心链路，再做体验增强。
- 每个任务默认包含四类子任务：组件实现、demo 页面、README、跨端自测。

## 迭代 1（表单基础闭环，1~2 周）

### 迭代 1 目标

- 完成表单核心输入能力，确保业务页面可录入、可提交、可校验。

### 迭代 1 组件任务（高优）

- [ ] `k-input`：收敛为正式主文件，清理 `copy/backup` 历史文件
- [ ] `k-textarea`
- [ ] `k-radio`
- [ ] `k-checkbox`
- [ ] `k-switch`
- [ ] `k-form`
- [ ] `k-form-item`

### 配套治理任务

- [ ] 首页组件状态同步（`completed/developing/planning` 与真实状态一致）
- [ ] `pages.json` 注册项与 demo 页面目录对齐
- [ ] 组件 README 模板统一（props / emits / slots / 主题变量）

### 迭代 1 DoD（完成标准）

- [ ] 组件 API 与文档一致，demo 可操作
- [ ] APP / WEB / MP 至少完成一次冒烟验证
- [ ] 不新增 `grid`、`gap`、`@media` 等红线样式

## 迭代 2（展示与反馈闭环，1~2 周）

### 迭代 2 目标

- 解决“列表承载 + 反馈提示”的页面主链路问题。

### 迭代 2 组件任务（高优）

- [ ] `k-list`
- [ ] `k-list-item`
- [ ] `k-popup`
- [ ] `k-modal`
- [ ] `k-toast`

### 增强任务（可并行）

- [ ] `k-swiper`
- [ ] `k-swiper-item`

### 迭代 2 DoD（完成标准）

- [ ] 列表场景可承载真实数据项（空态、长内容、边界项）
- [ ] 弹层与提示具备最小可用交互（开关、关闭、禁用态）
- [ ] 示例文案与组件行为一致

## 迭代 3（导航与滚动能力，1~2 周）

### 迭代 3 目标

- 完成应用级页面组织能力，形成可组合的基础页面骨架。

### 迭代 3 组件任务（高优）

- [ ] `k-tabs`
- [ ] `k-tab-pane`
- [ ] `k-navbar`
- [ ] `k-tabbar`
- [ ] `k-tabbar-item`
- [ ] `k-pull-refresh`
- [ ] `k-load-more`

### 迭代 3 DoD（完成标准）

- [ ] 导航组件可在 demo 中串联成完整页面流
- [ ] 下拉刷新与上拉加载具备可演示的状态切换
- [ ] 文档、示例、组件状态清单三者同步

## 暂不纳入一期（避免分散产能）

### 业务耦合型组件

- `k-goods-card`、`k-goods-list`、`k-goods-detail`
- `k-cart`、`k-submit-bar`
- `k-address-picker`、`k-address-list`、`k-address-edit`
- `k-user-card`、`k-comment`

### 低复用或沉底能力

- `k-link`（可由 `k-text` 承担）
- `k-copy`、`k-share`（建议沉到 `utils`）
- `k-scroll-view`（优先使用平台原生能力）

### 高复杂预研类（二期评估）

- 图表、地图、富文本编辑类组件

## Issue 模板（建议）

```base
标题：feat(组件): 新增 k-xxx 组件（一期）

验收标准：
- [ ] 组件 API 定义完成（props/emits/slots）
- [ ] demo 页面已创建并注册 pages.json
- [ ] README 已补齐并与实现一致
- [ ] 完成 APP / WEB / MP 冒烟验证
- [ ] 满足 kit-ui 一期 rules 红线要求
```
